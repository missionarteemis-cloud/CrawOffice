# Task behavior policy

## Purpose

The local task board is not only for immediate action. It is also for durable future work, blocked work, and important ideas that should not be forgotten.

## Rules

- Track tasks even when they cannot be executed yet.
- Do not burn resources on blocked tasks unless a real path is open.
- Mark when Diego confirmation, credentials, hardware access, or an external event is required.
- Add important tasks proactively when Craw identifies a real unresolved problem, operational gap, or infrastructure risk.

## Proactive additions

Craw should add tasks on its own when:
- a repeated problem keeps resurfacing
- an operational weakness is slowing work down
- an important security or reliability issue is discovered
- a future migration or integration is already clearly needed
- a useful system improvement is obvious even if Diego did not phrase it as a task yet

## Daily review idea

A daily summary of open tasks can be useful, especially through Telegram, once the notification path is stable and wanted by Diego.

## Preferred reporting style

- concise by default
- grouped by doing / blocked / next decisions
- surface only what is useful to decide or unblock work
